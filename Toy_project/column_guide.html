04_preprocess_team_update.py 기준 · 태그: 원본 / 파생 / 신규 / 누수주의 / 원-핫

1. 제품 기본 정보 (product CSV에서 옴)
product_id원본
KREAM에서 각 신발에 부여한 고유 식별 번호.
모든 집계·조인의 기준 키야. 숫자지만 의미 있는 값은 아니고 그냥 구분자야.
release_date원본
해당 제품이 공식적으로 처음 출시된 날짜.
출시 직후인지, 한참 지난 후인지 판단하는 기준이 돼. days_since_release 같은 파생 피처의 기반이야.
release_price원본
브랜드가 정한 공식 출시 가격 (단위: 원).
프리미엄 비율 계산의 분모야. 출시가가 높을수록 프리미엄 배수가 낮게 나오는 구조야.
is_collaboration원본
유명 브랜드·아티스트와의 콜라보레이션 제품 여부. 맞으면 1, 아니면 0.
콜라보 제품은 희소성이 높아서 프리미엄이 강하게 형성되는 경향이 있어. 중요한 범주형 피처야.
color_white / color_black / ... (11개)원-핫
색상 문자열 → COLOR_SYNONYMS 동의어 테이블로 매핑 → 0 또는 1 예) "오프화이트" → color_white = 1 "네이비" → color_navy = 1 복합색이면 여러 컬럼 동시에 1 가능
제품 색상을 11개 카테고리(white·black·grey·brown·pink·blue·navy·green·red·orange·yellow)로 인코딩한 것.
색상에 따라 수요가 달라지기 때문에 피처로 포함했어. 검정·흰색 같은 기본 색상은 대체로 수요가 안정적이고, 특수 색상은 희소성이 높아지는 경향이 있어.
wish_count누수주의
KREAM에서 해당 제품을 위시리스트에 추가한 사람 수. 단, 이 값은 크롤링 시점(2026-02-19) 기준이야.
관심도의 절대적인 크기를 나타내. 값이 클수록 많은 사람이 원하는 제품이야.
⚠️ 미래 정보 주의 — 모델링 시 직접 피처로 쓰면 안 돼. adjusted_wish로 보정해서 써야 해.
2. 사이즈
size원본
해당 거래의 신발 사이즈 (단위: mm).
같은 제품이라도 사이즈에 따라 수요·프리미엄이 크게 다를 수 있어.
golden_size파생
235 ≤ size ≤ 245 → 1 (여성 주요 사이즈대) 265 ≤ size ≤ 275 → 1 (남성 주요 사이즈대) 그 외 → 0
수요가 집중되는 황금 사이즈 여부.
황금 사이즈는 구매 수요가 많아서 프리미엄이 더 높게 형성되는 경향이 있어. 희소성과 수요가 동시에 높은 구간이야.
3. 거래 기본 정보 (trade CSV에서 옴)
first_trade_date파생
first_trade_date = product_id별 trade_date의 최솟값
해당 제품의 첫 번째 거래가 발생한 날짜.
제품이 2차 시장에서 언제부터 거래되기 시작했는지를 나타내. days_since_first_trade 등 시간 기반 피처의 기준점이야.
days_since_first_trade파생
days_since_first_trade = (trade_date - first_trade_date).days
첫 거래일로부터 현재 거래일까지 경과한 일수.
0이면 첫 거래 당일이야. 값이 작을수록 출시 초기 거래고, 클수록 시간이 많이 지난 거래야. 시간에 따른 프리미엄 변화 패턴을 잡는 핵심 피처야.
trade_date원본
해당 거래가 실제로 발생한 날짜.
각 행의 시점을 나타내는 기준이야. 모델링에서는 직접 피처로 쓰지 않고, 파생 피처 계산에 쓰여.
current_price원본
해당 거래에서 실제로 체결된 가격 (단위: 원).
2차 시장에서 소비자가 실제로 지불한 금액이야. premium_ratio 계산의 핵심 재료야.
premium_ratio파생
premium_ratio = (current_price - release_price) / release_price
출시가 대비 현재 거래가의 프리미엄 비율.
예) 출시가 10만 원짜리가 15만 원에 팔리면 premium_ratio = 0.5 (50% 프리미엄).
음수면 출시가보다 낮게 팔린 거야.
이 값이 K-Means 클러스터링의 입력이자 모델의 타겟(y)을 만드는 기반이야. current_price와 release_price 모두 해당 거래 시점에 알 수 있는 값이라 누수가 아니야.
cum_trade_count_product파생
cum_trade_count_product = 제품별 거래 행의 순서 번호 (1부터 시작) → groupby(product_id).cumcount() + 1
해당 거래까지 누적으로 몇 번째 거래인지를 나타내는 값.
예) 어떤 제품의 3번째 거래면 값이 3.
시간이 지날수록 누적 거래량이 많아지므로, 제품의 전반적인 인기·유동성을 나타내는 간접 지표야.
4. 구글 트렌드
google_trend_release원본
제품 출시 시점 전후의 구글 검색 트렌드 지수 (0~100).
출시 당시 얼마나 화제가 됐는지를 나타내. 출시 시점 관심도가 높을수록 초기 프리미엄이 높게 형성되는 경향이 있어.
google_trend_n_day원본
해당 거래 시점의 구글 검색 트렌드 지수 (0~100).
거래가 이뤄지는 시점에도 여전히 관심이 높은지를 나타내. 출시 후에도 검색이 유지된다는 건 지속적인 수요가 있다는 신호야.
trend_retention파생
trend_retention = google_trend_n_day / (google_trend_release + 1)
출시 시점 트렌드 대비 거래 시점 트렌드가 얼마나 유지되고 있는지의 비율.
값이 1이면 출시 때와 똑같은 관심도, 1보다 크면 오히려 더 올라간 것, 1보다 작으면 관심이 식은 것.
분모에 +1을 더한 건 google_trend_release가 0일 때 0으로 나누는 걸 방지하기 위해서야.
관심도가 얼마나 지속되는지를 나타내. 트렌드가 유지될수록 프리미엄도 유지되는 경향이 있어.
adjusted_google_trend파생
adjusted_google_trend = (google_trend_release × google_trend_n_day) / (google_trend_release + google_trend_n_day + 1)
출시 트렌드와 거래 시점 트렌드의 조화 평균 방식 결합값.
두 값이 모두 높아야 큰 값이 나오는 구조야. 한쪽만 높으면 값이 작아져.
예) release=80, n_day=10이면 ≈ 10.8 (거래 시점 관심이 낮으니 낮게 평가됨)
예) release=80, n_day=70이면 ≈ 37.3 (둘 다 높으니 높은 값)
출시 때도 화제가 되고 거래 시점에도 여전히 관심이 있는 제품을 높게 평가하는 피처야.
5. Wish 관련
wish_per_day누수주의
denom_days = (COLLECTION_DATE - first_trade_date).days wish_per_day = wish_count / denom_days
첫 거래일부터 크롤링 시점까지 하루 평균 wish가 얼마나 쌓였는지를 나타내는 값.
adjusted_wish를 계산하기 위한 중간 변수야. CSV에는 포함되지만 모델 피처로 직접 쓰면 안 돼.
wish_count 기반이라 미래 정보야. 직접 학습에 쓰지 말 것.
adjusted_wish파생
adjusted_wish = wish_per_day × (days_since_first_trade + 1) = wish_count × (days_since_first_trade + 1) / (COLLECTION_DATE - first_trade_date).days 가정: wish_count는 첫 거래일부터 크롤링까지 매일 균등하게 증가한다.
균등 증가 가정으로 해당 거래 시점의 wish 수를 추정한 값.
예) 크롤링까지 총 100일, wish_count=1000, 거래일이 50일 경과했다면 → 500으로 추정.
완전한 해당 시점 값은 아니지만 wish_count를 직접 쓰는 것보다는 누수가 완화돼.
해당 시점의 관심도를 근사한 값이야. 보조 피처로 활용할 수 있어.
6. 수요 강도 대리 피처 (신규 추가)
trade_velocity신규
trade_velocity = cum_trade_count_product / (days_since_first_trade + 1)
첫 거래 이후 하루 평균 누적 거래 수. 분모에 +1을 더해 첫 거래 당일(0일)의 0 나눗셈을 방지해.
예) 10일 동안 거래가 30건 쌓였으면 → 30 / 11 ≈ 2.7
거래가 빠르게 쌓이는 제품일수록 수요가 강하다는 신호야. wish 없이 수요 강도를 나타내는 핵심 대리 피처야.
trend_x_trade신규
trend_x_trade = adjusted_google_trend × trade_velocity
구글 트렌드(검색 관심)와 거래 속도를 곱한 상호작용 피처.
검색 관심도 높고 + 거래도 빠르게 쌓이는 제품만 큰 값이 나와.
검색 관심이 실제 거래로도 이어지고 있는지를 동시에 포착해. 관심만 많고 거래가 없거나, 거래는 있어도 검색이 없으면 값이 낮아져.
early_trade_signal신규
early_trade_signal = (days_since_first_trade ≤ 7) × trade_velocity → 첫 거래일 기준 7일 이내 거래면 trade_velocity 값 그대로, 아니면 0
출시 초기 7일 이내의 거래 속도를 별도로 포착한 피처.
초기 7일이 지난 거래는 모두 0이 돼.
출시 직후 폭발적인 수요가 있었는지를 나타내. 초기에 거래가 몰린 제품은 이후에도 높은 프리미엄을 유지하는 경향이 있어.
price_std_product신규
price_std_product = 제품별 누적 거래가격의 표준편차 (expanding std) → groupby(product_id)['current_price'].expanding().std()
해당 거래까지 누적된 거래가격들의 표준편차.
첫 거래는 NaN (비교 대상이 1개뿐이라 표준편차 계산 불가), 두 번째 거래부터 계산돼.
가격이 얼마나 들쑥날쑥한지를 나타내. 표준편차가 크면 프리미엄이 불안정하다는 신호이고, 작으면 거래가가 일정하게 유지된다는 뜻이야.
trade_growth_7d신규
daily_trade_count = product_id × trade_date별 하루 거래 건수 recent_7d = 최근 7일 거래량 합산 (rolling 7) prev_7d = 그 이전 7일 거래량 합산 (rolling 14 - recent_7d) trade_growth_7d = log(recent_7d + 1) - log(prev_7d + 1)
최근 7일 거래량이 이전 7일 대비 얼마나 증가했는지를 로그 스케일로 표현한 값.
양수면 거래 가속 중, 음수면 거래 감속 중, 0에 가까우면 비슷한 수준을 유지 중이야.
로그를 쓴 이유는 거래량이 갑자기 폭발적으로 늘어날 때 값이 너무 커지는 걸 방지하기 위해서야.
거래 모멘텀을 나타내. 거래가 가속되고 있으면 프리미엄도 오를 가능성이 높아.
7. 추가 파생 피처 (신규 추가)
days_since_release신규
days_since_release = (trade_date - release_date).days
공식 출시일로부터 해당 거래일까지 경과한 일수.
days_since_first_trade와 비슷하지만 기준점이 달라. first_trade_date는 2차 시장에서 첫 거래가 발생한 날이고, release_date는 브랜드의 공식 출시일이야.
출시 후 얼마나 됐는지를 공식 출시 기준으로 보는 피처야. 출시 직후일수록 초기 프리미엄이 높게 형성되는 경향이 있어.
hist_avg_premium신규
hist_avg_premium = 직전 거래까지의 premium_ratio 누적 평균 → groupby(product_id)['premium_ratio'].transform( lambda x: x.shift(1).expanding().mean() ).fillna(0) shift(1): 현재 거래의 premium_ratio는 제외 (미래 정보 방지) expanding(): 직전 거래까지 누적 평균 fillna(0): 첫 거래는 이전 값이 없으므로 0으로 대체
현재 거래 이전까지의 역대 평균 프리미엄 비율.
예) 이전에 5번 거래됐고 프리미엄이 0.3, 0.4, 0.5, 0.4, 0.3이었다면 hist_avg_premium = 0.38
이 제품이 지금까지 어느 정도의 프리미엄을 유지해왔는지를 나타내. 과거 프리미엄이 높은 제품은 앞으로도 높게 유지될 가능성이 있어.
rolling_price_std신규
rolling_price_std = 직전 거래 기준 최근 7건 거래가격의 표준편차 → groupby(product_id)['current_price'].transform( lambda x: x.shift(1).rolling(window=7, min_periods=2).std() ).fillna(0) shift(1) : 현재 거래 가격 제외 rolling(7) : 직전 최대 7건 기준 min_periods=2: 최소 2건 있어야 계산 (1건이면 표준편차 불가) fillna(0) : 데이터 부족하면 0 처리
직전 최대 7건 거래가격의 단기 가격 변동성.
price_std_product(전체 누적)와 달리 최근 7건만 보는 단기 버전이야.
최근 거래가격이 얼마나 흔들리는지를 나타내. 값이 크면 가격이 불안정한 것이고, 작으면 최근 가격이 일정하게 유지되고 있다는 뜻이야.
trade_interval_density신규
거래 간격 = 현재 거래일 - 직전 거래일 (일수) trade_interval_density = 직전 거래 기준 최근 3건 거래 간격의 합산 → diff().dt.days : 각 거래 간의 간격(일수) 계산 shift(1) : 현재 거래 간격 제외 rolling(window=3) : 직전 최대 3건의 간격 합산 fillna(0) : 첫 거래는 0 처리 ※ 이름 주의: 기존 trade_velocity(누적 거래 속도)와 다른 피처야.
최근 3건 거래 간격의 합산 일수. 값이 작을수록 거래가 짧은 간격으로 밀집되고 있다는 뜻이야.
예) 최근 거래 간격이 1일, 2일, 1일이었다면 → 4 (거래 매우 밀집)
예) 최근 거래 간격이 10일, 15일, 20일이었다면 → 45 (거래 드문드문)
짧은 간격으로 거래가 몰리고 있다는 건 수요가 급증하고 있다는 신호야. 특히 최근 거래 활성화 여부를 포착할 수 있어.
8. 데이터 품질
status파생
이상값이 없으면: "-" 있으면 해당하는 태그를 공백으로 연결: [premium_ratio=inf] → current_price / release_price 계산 오류 [adjusted_wish=NaN] → wish_count 또는 날짜 계산 실패 [google_trend_*=NaN] → 트렌드 데이터 누락 [current_price<0] → 음수 거래가 (이상값)
해당 행에 데이터 품질 이슈가 있는지를 기록한 텍스트 컬럼.
모델링 전에 이상한 행을 찾아내거나 필터링할 때 활용해. 정상 데이터는 "-"로 표시돼.
9. 내부 계산용 컬럼 (모델링에서 제외 권장)
trade_date_dt / release_date_dt파생
trade_date, release_date를 datetime 타입으로 변환한 컬럼. days_since_release 등 날짜 계산에만 사용하고, 원본 문자열 컬럼과 중복돼.
모델링 시 drop_cols에 넣어서 제외하는 게 좋아.
