1
제품 기본 정보

product CSV에서 가져오는 컬럼들입니다. 제품 자체의 속성을 나타냅니다.

product_id원본
KREAM 플랫폼이 각 신발에 부여한 고유 식별 번호.
모든 집계·조인의 기준 키야. 숫자이지만 의미는 없고, 그냥 제품 구분자야. 외부 검색 불가.
release_date원본
해당 제품이 공식적으로 처음 출시된 날짜.
💡 오래된 제품은 KREAM 데이터 누락이 많아, 기록이 없으면 첫 거래일을 발매일로 대체했어.
days_since_release 등 시간 기반 파생 피처의 기준점이야.
release_price원본
브랜드가 정한 공식 출시 가격 (단위: 원).
프리미엄 비율의 분모야. 출시가가 높을수록 같은 거래가에서 프리미엄 비율이 낮게 나와.
is_collaboration원본
유명 브랜드·아티스트와의 콜라보레이션 여부. 맞으면 1, 아니면 0.
콜라보 제품은 희소성이 높아 프리미엄이 강하게 형성되는 경향이 있어. 중요한 범주형 피처야.
color_white / color_black / ... (11개) 원-핫
색상 문자열 → COLOR_SYNONYMS 동의어 테이블로 매핑 → 0 또는 1 예) "오프화이트" → color_white = 1 "네이비" → color_navy = 1 복합색이면 여러 컬럼 동시에 1 가능 11개 색상: white / black / grey / brown / pink / blue / navy / green / red / orange / yellow ↑ Red와 Yellow를 통합하지 않은 이유: 실제 데이터에서 Red는 프리미엄이 오르고 Yellow는 떨어지는 패턴이 달랐기 때문.
제품 색상을 11개 카테고리로 인코딩한 것.
색상에 따라 수요가 달라지기 때문에 피처로 포함했어. 기본 색상(검정·흰색)은 수요가 안정적이고, 특수 색상은 희소성이 높아.
wish_count누수주의
KREAM에서 해당 제품을 위시리스트에 추가한 사람 수. 크롤링 시점(2026-02-19) 기준이야.
관심도의 절대적인 크기를 나타내. 값이 클수록 많은 사람이 원하는 제품이야.
크롤링 시점의 총합 수치라 미래 정보가 포함돼 있어. 직접 피처로 쓰면 안 되고, adjusted_wish로 보정해서 써야 해.
2
사이즈
size원본
해당 거래의 신발 사이즈 (단위: mm).
같은 제품이라도 사이즈에 따라 수요·프리미엄이 크게 달라질 수 있어.
golden_size파생
235 ≤ size ≤ 245 → 1 (여성 주요 사이즈대) 265 ≤ size ≤ 275 → 1 (남성 주요 사이즈대) 그 외 → 0
수요가 집중되는 황금 사이즈 여부.
황금 사이즈는 구매 수요가 많아서 프리미엄이 더 높게 형성되는 경향이 있어. 희소성과 수요가 동시에 높은 구간이야.
3
거래 기본 정보

trade CSV에서 가져오거나, 거래 데이터로부터 계산한 컬럼들입니다.

trade_date원본
해당 거래가 실제로 발생한 날짜.
각 행의 시점을 나타내는 기준이야. 모델링에서는 직접 피처로 쓰지 않고, 파생 피처 계산에 쓰여.
current_price원본
해당 거래에서 실제로 체결된 가격 (단위: 원).
2차 시장에서 소비자가 실제로 지불한 금액이야. premium_ratio 계산의 핵심 재료야.
premium_ratio타겟 원천
premium_ratio = (current_price - release_price) / release_price 예) 출시가 10만 원짜리가 15만 원에 팔리면 → premium_ratio = 0.5 (50% 프리미엄) 음수면 출시가보다 낮게 팔린 것 (역프리미엄)
출시가 대비 현재 거래가의 프리미엄 비율.
이 값이 K-Means 클러스터링의 입력이자, 모델의 타겟(y)을 만드는 기반이야. current_price와 release_price 모두 해당 거래 시점에 알 수 있는 값이라 누수가 아니야.
first_trade_date파생
first_trade_date = product_id별 trade_date의 최솟값
해당 제품의 첫 번째 거래가 발생한 날짜.
제품이 2차 시장에서 언제부터 거래되기 시작했는지를 나타내. days_since_first_trade 등 시간 피처의 기준점이야.
days_since_first_trade파생
days_since_first_trade = (trade_date - first_trade_date).days
첫 거래일로부터 현재 거래일까지 경과한 일수.
0이면 첫 거래 당일, 값이 클수록 시간이 많이 지난 거래야.
시간에 따른 프리미엄 변화 패턴을 잡는 핵심 피처야.
cum_trade_count_product파생
cum_trade_count_product = 제품별 거래 행의 순서 번호 (1부터 시작) → groupby(product_id).cumcount() + 1
해당 거래까지 누적으로 몇 번째 거래인지를 나타내는 값.
시간이 지날수록 누적 거래량이 많아지므로, 제품의 전반적인 인기·유동성을 나타내는 간접 지표야.
4
구글 트렌드

구글 트렌드 CSV(검색어: "Nike", 2019.07~2026.02)에서 매핑한 값들입니다.

📌
구글 트렌드 사용 방식 구글 트렌드는 기간 내 상대지수(0~100)야. 제품별 정확한 키워드 수집이 어려워 "Nike" 단일 키워드로 2019.07~2026.02 전체를 수집했어. 덕분에 제품 간 비교가 가능한 공통 기준이 생겼지만, 개별 신발의 정확한 관심도는 아니야. google_trend_pre는 수집 기간 한계로 최종 제외됐어.
google_trend_release원본
제품 출시 시점의 구글 검색 트렌드 지수 (0~100).
출시 당시 얼마나 화제가 됐는지를 나타내. 출시 시점 관심도가 높을수록 초기 프리미엄이 높게 형성되는 경향이 있어.
google_trend_n_day원본
해당 거래 시점의 구글 검색 트렌드 지수 (0~100).
거래가 이뤄지는 시점에도 여전히 관심이 높은지를 나타내. 출시 후에도 검색이 유지된다는 건 지속적인 수요가 있다는 신호야.
trend_retention파생
trend_retention = google_trend_n_day / (google_trend_release + 1) 분모에 +1: google_trend_release가 0일 때 0÷0을 방지하기 위해서야. 값 해석: > 1.0 → 출시 때보다 오히려 관심이 올라간 것 = 1.0 → 출시와 똑같은 관심도 유지 < 1.0 → 관심이 식은 것
출시 시점 트렌드 대비 거래 시점 트렌드가 얼마나 유지되는지의 비율.
관심도가 얼마나 지속되는지를 나타내. 트렌드가 유지될수록 프리미엄도 유지되는 경향이 있어.
adjusted_google_trend파생
adjusted_google_trend = (google_trend_release × google_trend_n_day) / (google_trend_release + google_trend_n_day + 1) 원리: 조화평균 — 두 값이 모두 높아야 큰 값이 나오는 구조 예) release=80, n_day=10 → ≈ 8.9 (거래 시점 관심 낮음 → 낮게 평가) release=80, n_day=70 → ≈ 37.3 (둘 다 높음 → 높게 평가)
출시 트렌드와 거래 시점 트렌드의 조화 평균 방식 결합값.
출시 때도 화제가 되고, 거래 시점에도 여전히 관심이 있는 제품을 높게 평가하는 피처야. 한쪽만 높으면 값이 작아져.
5
Wish 관련
wish_per_day누수주의
denom_days = (COLLECTION_DATE - first_trade_date).days ← 2026-02-19 기준 wish_per_day = wish_count / denom_days
첫 거래일부터 크롤링 시점까지 하루 평균 wish가 얼마나 쌓였는지를 나타내는 중간 계산 변수야.
wish_count 기반이라 미래 정보야. 직접 학습 피처로 쓰지 말 것. adjusted_wish 계산에만 사용.
adjusted_wish파생
adjusted_wish = wish_per_day × (days_since_first_trade + 1) = wish_count × (days_since_first_trade + 1) / (COLLECTION_DATE - first_trade_date).days 핵심 가정: wish_count는 첫 거래일부터 크롤링까지 매일 균등하게 증가한다. 예) 크롤링까지 총 100일, wish_count=1000, 거래일이 50일 경과 → 500으로 추정
균등 증가 가정으로 해당 거래 시점의 wish 수를 추정한 값.
완전한 해당 시점 값은 아니지만 wish_count 직접 사용보다는 누수가 완화돼.
해당 시점의 관심도를 근사한 값이야. 보조 피처로 활용할 수 있어.
6
수요 강도 대리 피처 신규 추가

이번 전처리에서 새로 추가된 피처들입니다. wish 없이 수요 강도를 포착하기 위해 만들었어요.

trade_velocity신규
trade_velocity = cum_trade_count_product / (days_since_first_trade + 1) 분모에 +1: 첫 거래 당일(0일)의 0 나눗셈을 방지 예) 10일 동안 거래가 30건 쌓였으면 → 30 / 11 ≈ 2.7
첫 거래 이후 하루 평균 누적 거래 수. 거래 속도를 나타내는 핵심 지표야.
거래가 빠르게 쌓이는 제품일수록 수요가 강하다는 신호야. wish 없이 수요 강도를 나타내는 핵심 대리 피처야.
trend_x_trade신규
trend_x_trade = adjusted_google_trend × trade_velocity
구글 트렌드(검색 관심)와 거래 속도를 곱한 상호작용 피처.
검색 관심도 높고 + 거래도 빠르게 쌓이는 제품만 큰 값이 나와.
검색 관심이 실제 거래로도 이어지고 있는지를 동시에 포착해. 관심만 많고 거래가 없거나, 거래는 있어도 검색이 없으면 값이 낮아져.
early_trade_signal신규
early_trade_signal = (days_since_first_trade ≤ 7) × trade_velocity → 첫 거래일 기준 7일 이내 거래면 trade_velocity 값 그대로 → 7일이 지난 거래는 모두 0
출시 초기 7일 이내의 거래 속도를 별도로 포착한 피처야.
출시 직후 폭발적인 수요가 있었는지를 나타내. 초기에 거래가 몰린 제품은 이후에도 높은 프리미엄을 유지하는 경향이 있어.
price_std_product신규
price_std_product = 제품별 누적 거래가격의 표준편차 (expanding std) → groupby(product_id)['current_price'].expanding().std() 첫 거래는 NaN (비교 대상 1개 → 표준편차 계산 불가), 두 번째 거래부터 계산됨
해당 거래까지 누적된 거래가격들의 표준편차.
가격이 얼마나 들쑥날쑥한지를 나타내. 표준편차가 크면 프리미엄이 불안정하다는 신호, 작으면 거래가가 일정하게 유지된다는 뜻이야.
trade_growth_7d신규
daily_trade_count = product_id × trade_date별 하루 거래 건수 recent_7d = 최근 7일 거래량 합산 (rolling 7) prev_7d = 그 이전 7일 거래량 합산 (rolling 14 합산 - recent_7d) trade_growth_7d = log(recent_7d + 1) - log(prev_7d + 1) 로그를 쓰는 이유: 거래량이 갑자기 폭발적으로 늘어날 때 값이 너무 커지는 걸 방지 양수 = 거래 가속 중 / 음수 = 거래 감속 중 / 0 근처 = 유사한 수준 유지
최근 7일 거래량이 이전 7일 대비 얼마나 증가했는지를 로그 스케일로 표현한 값.
거래 모멘텀을 나타내. 거래가 가속되고 있으면 프리미엄도 오를 가능성이 높아.

추가 파생 피처 신규 추가
days_since_release신규
days_since_release = (trade_date - release_date).days ※ days_since_first_trade와 비교: days_since_first_trade : 2차 시장 첫 거래일 기준 days_since_release : 브랜드 공식 출시일 기준 (더 이른 날짜)

출시 후 얼마나 됐는지를 공식 출시 기준으로 보는 피처야. 출시 직후일수록 초기 프리미엄이 높게 형성되는 경향이 있어.
hist_avg_premium신규
hist_avg_premium = 직전 거래까지의 premium_ratio 누적 평균 → groupby(product_id)['premium_ratio'].transform( lambda x: x.shift(1).expanding().mean() ).fillna(0) shift(1) : 현재 거래의 premium_ratio는 제외 (미래 정보 방지!) expanding() : 직전 거래까지 누적 평균 fillna(0) : 첫 거래는 이전 값이 없으므로 0으로 대체
현재 거래 이전까지의 역대 평균 프리미엄 비율.
예) 이전에 5번 거래됐고 프리미엄이 0.3, 0.4, 0.5, 0.4, 0.3이었다면 hist_avg_premium = 0.38
이 제품이 지금까지 어느 정도의 프리미엄을 유지해왔는지를 나타내. 과거 프리미엄이 높은 제품은 앞으로도 높게 유지될 가능성이 있어.
rolling_price_std신규
rolling_price_std = 직전 거래 기준 최근 7건 거래가격의 표준편차 → groupby(product_id)['current_price'].transform( lambda x: x.shift(1).rolling(window=7, min_periods=2).std() ).fillna(0) price_std_product(전체 누적)와 달리, 이건 최근 7건만 보는 단기 버전
직전 최대 7건 거래가격의 단기 가격 변동성.
최근 거래가격이 얼마나 흔들리는지를 나타내. 값이 크면 가격이 불안정한 것, 작으면 최근 가격이 일정하게 유지되고 있다는 뜻이야.
trade_interval_density신규
거래 간격 = 현재 거래일 - 직전 거래일 (일수) trade_interval_density = 직전 거래 기준 최근 3건 거래 간격의 합산 일수 예) 최근 거래 간격이 1일, 2일, 1일 → 값 = 4 (거래 매우 밀집) 최근 거래 간격이 10일, 15일, 20일 → 값 = 45 (거래 드문드문) ※ trade_velocity(누적 거래 속도)와 다른 피처야 — 이름 혼동 주의!
최근 3건 거래 간격의 합산 일수. 값이 작을수록 거래가 밀집되어 있다는 뜻이야.
짧은 간격으로 거래가 몰리고 있다는 건 수요가 급증하고 있다는 신호야. 특히 최근 거래 활성화 여부를 포착할 수 있어.

S1
Step 1: 타겟 라벨 생성 — K-Means 클러스터링
❓
왜 클러스터링으로 타겟을 만드나? premium_ratio는 연속형 숫자야. 그런데 우리 목표는 "이 신발이 프리미엄 등급 몇 단계냐?"를 맞추는 분류 문제야. 사전에 정해진 정답 등급이 없으니, 데이터 자체의 분포를 K-Means로 자동 분할해서 의미 있는 등급(낮음/보통/높음/...)을 만드는 거야.
항목내용이유
입력값log1p(premium_ratio) — log 변환음수 비율 처리 + 분포 정규화
K 탐색 범위K = 2 ~ 6너무 많으면 의미 없고, 너무 적으면 구분력 떨어져
K 선택 기준Elbow Method + Silhouette Score 동시 확인두 지표가 같은 K를 가리킬수록 신뢰도 높음
라벨 순서평균 premium_ratio 기준 오름차순 재매핑 (0~4)등급 번호가 실제 프리미엄 크기와 대응되도록
누수 방지KMeans는 Train 데이터로만 fit → Test는 predict만Test 라벨이 학습에 영향을 주지 않도록
📣
Elbow Curve와 Silhouette 그래프를 나란히 보여주며 "왜 이 K를 선택했는가"를 설명하세요. 두 지표가 같은 K를 가리킬수록 설득력이 높습니다. 클러스터별 프리미엄 범위 박스플롯도 함께 보여주면 각 등급의 의미가 직관적으로 전달됩니다.
S2
Step 2: 데이터 분할 — 제품 단위 GroupSplit
⚠️
왜 제품 단위로 분리하나? — 데이터 누수 방지 동일 제품의 여러 거래(다른 n일 값)가 train/test에 섞이면, 제품 고유 특성(release_price, color 등)이 test에 간접 노출됩니다. 이렇게 되면 모델이 "본 적 있는 제품"을 맞추는 게 돼서 성능이 과대평가됩니다.
항목내용
분할 방식GroupShuffleSplit(group=product_id)
비율Train 약 96개 제품 (80%) / Test 약 24개 제품 (20%)
random_state 선택발매가 구간별 비율이 가장 균등한 split 선택
교차검증Train 내부에서 GroupKFold 5-fold로 하이퍼파라미터 튜닝
S3
Step 3: Model 1 — Baseline XGBoost
📖
왜 XGBoost를 쓰나? Kaggle 등 실전 데이터 경쟁에서 반복적으로 검증된 모델이야. 비선형 관계를 잘 잡고, 피처 중요도 추출이 쉬우며, 다중 분류(multi:softmax)를 기본 지원해.
하이퍼파라미터값설정 이유
n_estimators500과적합 없이 충분한 앙상블 크기 확보
learning_rate0.05낮은 학습률 + 높은 트리 수 조합으로 안정성 확보
objectivemulti:softmax다중 분류 전용 목적함수
eval_metricmlogloss다중 분류 손실 지표
sample_weightbalanced (역빈도 가중치)소수 클래스(폭등) 학습 비중 보정
🤔
SMOTE를 쓰지 않은 이유 SMOTE는 소수 클래스 샘플을 인공적으로 생성해 데이터를 늘리는 방법이야. 그런데 우리 데이터는 제품 단위 특성이 강해서 인공 샘플이 실제 존재할 수 없는 조합을 만들어낼 수 있어. 노이즈가 늘어나는 셈이야. 그래서 sample_weight로 학습 가중치를 조절하는 방법을 선택했어.
📣
SMOTE를 쓰지 않은 이유를 발표에서 반드시 언급하세요. 단순히 몰라서가 아니라 데이터 특성을 고려한 판단임을 강조하면 평가단에게 좋은 인상을 줍니다.
S4
Step 4: TH4 후처리 — 소수 클래스 Recall 향상
🔧
왜 TH4 후처리가 필요한가? class 4(폭등)는 데이터가 매우 적어, 모델이 기본 argmax로 예측하면 class 4를 거의 예측하지 않아. Recall이 극도로 낮아지는 거야. 그래서 "class 4의 확률이 TH4 이상이면 강제로 class 4로 예측"하는 후처리를 추가했어.
항목내용
핵심 아이디어P(class=4) ≥ TH4 이면 강제로 class 4 예측
TH4 결정 방법Train 내부에서 별도 validation set 분리 후 자동 튜닝 (Test는 절대 참조 안 함)
Trade-offRecall 상승 대신 Precision 일부 감소 가능
적용 모델Model 1 + TH4 / Model 2 + TH4 모두에 동일한 TH4 값 적용
📣
TH4 후처리의 효과를 Confusion Matrix Before/After로 나란히 비교하면 시각적으로 강한 임팩트를 줍니다. class 4 행(row)에서 Recall이 얼마나 올랐는지를 숫자로 강조하세요.
S5
Step 5: Feature Selection — VIF + Feature Importance
🔍
왜 Feature Selection을 하나? 피처가 많다고 모델이 좋아지지 않아. 노이즈 피처가 섞이면 오히려 모델이 중요한 신호를 놓칠 수 있어. 또 VIF가 높은 피처끼리는 같은 정보를 중복 제공하는 거라, 제거하면 모델이 실제로 중요한 신호에 집중할 수 있어.
방법기준역할
Feature Importance (Gain)Model 1 학습 결과에서 추출중요도 낮은 피처 제거 후보 선정
VIF (분산팽창지수)VIF ≥ 10: 다중공선성 높음 → 제거중복 정보 제거 — 높은 VIF 피처부터 순차 제거 후 반복
📐
Gain 방식이란? 해당 피처가 분기(split)에 사용될 때 평균적으로 가져다주는 불순도 감소량이야. 이 값이 크다는 건, 그 피처로 데이터를 나눌 때 target이 잘 분리된다는 뜻이야. → 실제로 예측에 도움이 되는 피처라는 의미.
📣
Feature Importance 바 차트를 Model 1 기준으로 먼저 보여주고, 어떤 피처가 살아남았는지 비교하세요. days_since_release, adjusted_google_trend 같이 직관적으로 이해되는 피처가 상위에 오면 설득력이 높아집니다.
S6
Step 6: Model 2 — Refined XGBoost
✅
Model 2의 기대 효과 Feature Selection으로 선별된 피처만 사용하여 동일 하이퍼파라미터로 재학습해. 노이즈 피처를 제거했으니, 일반화 성능이 향상되고 모델 해석도 쉬워지는 게 기대야.
